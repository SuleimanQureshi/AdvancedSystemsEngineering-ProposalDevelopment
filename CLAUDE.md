# Editor instructions for the AESPD Deliverables site

This file tells Claude Code (and any AI assistant) how to edit this repo
safely. Read it at the start of every session.

## Hard rules — read before editing

1. **Never run `git` commands.** Not `git add`, not `git commit`, not
   `git push`, `git checkout`, `git merge`, `git status`, `git diff`, or
   any other git subcommand. Also do not run `gh` write commands
   (`gh repo create`, `gh pr ...`, etc.). All git and remote operations
   are reserved for the human editor.
2. **Never edit anything under `source/`.** That folder holds the original
   `.docx`, kept as an immutable reference. Treat it as read-only.
3. **End every response that touches files with a `Files modified` list.**
   The editor uses this list to scope their commit. Format:
   ```
   Files modified:
   - docs/stages/2-system-concept/d1-foo.md
   - docs/assets/images/foo-figure-1.png
   ```
4. **Stop at staged changes.** When work is complete, leave files on disk
   and report what changed. Do not attempt to commit, branch, or push.

## File and naming conventions

- One deliverable per file. Filenames are kebab-case and prefixed with
  `d<N>-`, e.g. `d3-system-functional-requirements.md`.
- Stage folders use a number prefix: `1-requirements`, `2-system-concept`,
  `3-stage-synthesis`, `4-later-deliverables`.
- Every deliverable file starts with front matter:
  ```yaml
  ---
  title: System Functional Requirements
  stage: System Concept
  deliverable_id: stage2-d3
  status: draft        # draft | review | approved
  last_reviewed: 2026-05-23
  ---
  ```
- Update `last_reviewed` (today's date) whenever you make a substantive
  edit to a deliverable.

## Comments — the 40 inline notes from the source doc

The source `.docx` carries 40 review comments authored by Sadaf Shaikh
between April and May 2026. They are preserved as collapsible admonitions
placed directly under the paragraph they annotate.

Always use this exact format:

```markdown
??? note "Sadaf · 2026-05-06"
    write instructions on establishing links between actions/assets and
    requirements generated by SRD because "Generate SRD" doesn't
    automatically create these links
```

- Title is `Sadaf · <ISO date>` — author then middle-dot then YYYY-MM-DD.
- Use `???` (collapsed by default), not `!!!` (always open).
- Never replace a comment with a blockquote or an inline parenthetical.
- When folding a comment into the prose (Phase 2 work), delete the
  admonition entirely once its intent is reflected in the body text.

## Cross-references between deliverables

Use relative paths with `.md` extensions; MkDocs resolves them at build:

```markdown
See [Stakeholder list](../1-requirements/d1-stakeholders.md#power-vs-interest)
for the upstream input.
```

Never hard-code the live site URL (`https://...github.io/...`). Relative
links keep working in local previews and after any future repo rename.

## Tables

- 6 columns or fewer → plain markdown table.
- 7 to 12 columns → HTML `<table>` inside the markdown (Material wraps it
  in a horizontal-scroll container automatically).
- More than 12 columns or more than 50 rows → store the data as CSV in
  `docs/assets/data/<name>.csv` and embed it with the table-reader
  plugin:
  ```markdown
  {{ read_csv('assets/data/stakeholder-matrix.csv') }}
  ```
  CSV diffs are far easier to review than wide HTML.

## Diagrams

- Prefer Mermaid diagrams over bitmap screenshots. They live inline in the
  markdown and are diffable:
  ````markdown
  ```mermaid
  graph TD
    A[Stakeholder] --> B[Need]
    B --> C[Requirement]
  ```
  ````
- For diagrams that cannot be expressed in Mermaid, save the source
  (drawio, excalidraw) in `docs/assets/diagrams/` and the exported PNG in
  `docs/assets/images/`.
- Filenames are descriptive and kebab-case:
  `power-vs-interest-matrix.png`, not `figure-1.png`.

## Images

- All images live under `docs/assets/images/`.
- Reference them with relative paths from the page:
  ```markdown
  ![Power vs Interest matrix](../../assets/images/power-vs-interest-matrix.png)
  ```
- Provide alt text — it's the screen-reader caption and the fallback if
  the image fails to load.

## Local preview while editing

The editor will usually have these two terminal tabs open:
1. `mkdocs serve` — local site at `http://127.0.0.1:8000`, hot-reloads
   on every save.
2. `claude` — this session.

You can assume hot-reload is on. After saving a file, you don't need to
restart anything; the editor will see the change in their browser within
a second or two.

If `mkdocs serve` is not running, you can suggest the editor start it
but don't try to start it yourself unless asked — it occupies a terminal
and blocks subsequent commands.

## Publishing model (informational — you don't do this)

The editor publishes by merging a feature branch into `main` and pushing
`main`. The GitHub Action then deploys to GitHub Pages. You do not run
any of these steps — you only produce the file changes that the editor
then commits.
